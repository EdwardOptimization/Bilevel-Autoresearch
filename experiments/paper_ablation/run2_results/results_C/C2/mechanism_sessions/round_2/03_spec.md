## **Implementation Specification: Post‑Generation Value Adjuster**

**1. Mechanism name**  
`post_generation_adjuster`

**2. Implementation strategy**  
`new_helper_class`

**3. Target**  
New class `PostGenerationAdjuster` to be instantiated in `TrainRunner.__init__` and used in `run_iteration`.

**5. Step‑by‑step logic**  

For `adjust_proposal` method:

1. **Validation check**  
   - If `adjustment_enabled` is False → return `proposed_value`
   - If `param_type != "continuous"` → return `proposed_value`
   - If `should_adjust_param(param_name)` returns False → return `proposed_value`

2. **Get adjustment direction**  
   - Query `momentum_tracker` for momentum vector for `param_name`
   - If momentum exists and is consistent (same sign for last 2-3 iterations):
     - `direction = sign(momentum)` (1 for increase, -1 for decrease)
   - Else:
     - Get best recent config from `elite_pool`
     - Compare `proposed_value` to best value in elite pool
     - If `proposed_value` < best_value → `direction = 1` (move toward better)
     - Else → `direction = -1`

3. **Calculate adjustment magnitude**  
   - Get base step size from `step_calibrator.get_step_size(param_name)`
   - Apply momentum amplification if available:
     - `magnitude = base_step_size * (1.0 + abs(momentum))`
   - Clamp magnitude to `max_relative_adjustment * abs(proposed_value)`

4. **Compute adjusted value**  
   - `adjusted = proposed_value + (direction * magnitude)`
   - Round to appropriate precision (6 decimal places for floats)

5. **Validate and apply bounds**  
   - Get valid range from `SearchConfig` for `param_name`
   - If `adjusted` outside valid range:
     - If `fallback_to_original` → return `proposed_value`
     - Else → clamp to nearest bound
   - If `adjusted` is NaN or infinite → return `proposed_value`

6. **Log adjustment**  
   - Record: `(param_name, proposed_value, adjusted, direction, magnitude, iteration)`
   - Store in adjustment history for debugging

7. **Return value**  
   - Return `adjusted` if all checks pass, otherwise `proposed_value`

For `should_adjust_param` method:

1. Get parameter definition from `SearchConfig`
2. Check if parameter has `type == "float"` or `type == "int"` with range
3. Check if parameter is in allowlist (initial allowlist: `["weight_decay", "learning_rate", "beta1", "beta2", "dropout"]`)
4. Return True only if all conditions met

In `TrainRunner.__init__`:

In `TrainRunner.run_iteration` (after LLM generates proposals):

```python
def run_iteration(self, iteration: int) -> TrainResult:
    # ... existing code up to LLM proposal generation
    
    # After LLM generates candidate changes but before applying them:
    if hasattr(self, 'value_adjuster') and self.value_adjuster.adjustment_enabled:
        adjusted_changes = []
        for change in llm_proposed_changes:
            if change['type'] == 'modify_param':
                current_val = self._extract_current_value(change['param'])
                adjusted_val = self.value_adjuster.adjust_proposal(
                    param_name=change['param'],
                    proposed_value=change['value'],
                    current_value=current_val,
                    param_type=self._get_param_type(change['param']),
                    iteration=iteration,
                )
                if adjusted_val != change['value']:
                    logger.info(
                        f"Adjusted {change['param']}: {change['value']} → {adjusted_val}"
                    )
                    change['value'] = adjusted_val
            adjusted_changes.append(change)
        llm_proposed_changes = adjusted_changes
    
    # Continue with applying changes to code...
```

**Additional considerations**:

1. **Debug logging**: Add verbose logging mode to show adjustment decisions
2. **Metrics tracking**: Track how often adjustments are applied vs. reverted
3. **Progressive enablement**: Start with only 1-2 parameters (weight_decay, learning_rate), expand based on success
4. **Fallback mechanism**: If adjusted values consistently perform worse than original LLM values over 5 iterations, automatically disable adjustments for that parameter
5. **Configuration**: Make all parameters (enable/disable, max_adjustment, allowlist) configurable via `SearchConfig`
